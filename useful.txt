--- Kubernetes installation- ---
Since you cannot use Docker Desktop, the best way to run Kubernetes on a Mac is to use a lightweight virtual machine manager as a "bridge" for the Kubernetes cluster.
Colima is an open-source, lightweight alternative to Docker Desktop that includes built-in Kubernetes support
brew install kubectl
colima start --kubernetes // This creates a Linux VM that automatically runs a Kubernetes cluster for you

docker compose is all manual work. → manual scaling + manual routing
Kubernetes → auto scaling + auto discovery + auto load balancing

Kubernetes doesn’t use local folders—it uses images. Build the images required.
docker build -t auth-service:latest ./auth-service
docker build -t journal-service:latest ./journal-service
docker build -t api-gateway:latest ./api-gateway

| Thing      | Meaning         |
| ---------- | --------------- |
| Docker     | builds image    |
| Kubernetes | runs containers |
| Deployment file | manages pods and scaling   | 
| Service file   | load balances  and service discovery  |
| HPA file   | Horizontal Pod Autoscaler  |


In Kubernetes, a Cluster is the overall hardware/software environment where your applications live, consisting of at least one master node and multiple worker machines called Nodes.

Think of it like a restaurant: the Cluster is the entire building, the Deployment is the kitchen staff managing how many cooks are working, and the Service is the waiter who takes orders and brings food to the right table.

After deployment and service files are created, run the following command to tell Kubernetes to make the cluster match the state defined in your files
kubectl apply -f k8s/

kubectl get pods // equals number of replicas
kubectl get services // 1 

// Port forward (creates a tunnel from your computer to the k8s service) => maps Port 3001 on your Mac/PC (localhost) to Port 3001 on the Service inside the cluster.
kubectl port-forward service/auth-service 3001:3001

http://localhost:3001/auth/login
Call → http://auth-service

in case of errors in kubectl get pods, check logs:
kubectl logs <pod-name>
kubectl describe pod <pod-name>
